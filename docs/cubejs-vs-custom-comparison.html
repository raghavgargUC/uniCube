Cube.js vs Custom Service (BullMQ + Redis)

For: 100K pre-aggregations | 4-5 rows each | 1000 tenants | 20 MySQL hosts | dynamic per-tenant/facility conditions

Why Cube.js uses more resources than Custom + Redis

Cube.js (higher footprint)

  • Memory: Cube Store keeps Parquet chunks, column metadata, and catalog entries for every pre-aggregation table. The server also holds compiled semantic models (per tenant/app id), query planners, and caches. Even tiny tables pay fixed overhead per table.
  • Disk: Each pre-agg is stored as Parquet files plus Cube Store metadata. Small row counts still create files and directory structure; 100K tables means 100K logical objects on disk.
  • Build time (worst case): A refresh run must orchestrate many builds, coordinate with MySQL, write to Cube Store, and update indexes. Throughput is bounded by concurrency settings and Cube Store write path, not only by row count.
  • Startup: Cube.js loads drivers, may warm connections, and initializes Cube Store and the orchestrator. Heavier than a thin worker that only connects to Redis.

Custom + Redis (lower footprint)

  • Memory: Redis stores only the result payload (e.g. a few KB of JSON per key). No OLAP engine, no per-table Parquet catalog, no compiled cube graph for 100K logical models.
  • Disk: Redis persistence (if enabled) is compact strings or snapshots; no per-pre-agg Parquet file layout.
  • Build time: A “build” is one SQL execution + SET in Redis. No second write layer after MySQL except cache update.
  • Startup: Workers are stateless; they connect to Redis and MySQL and start processing jobs quickly.

Chart note: Bar values are illustrative order-of-magnitude for comparing the two approaches under ~100K cached units and small result sets. Actual numbers depend on hardware, Cube version, Redis mode, and tuning. The point is structural: Cube.js adds an analytics layer (semantic model + columnar store); Custom + Redis adds a thin cache on top of MySQL.

Verdict for Your Use Case

Custom service wins for the alert/subscription system (100K dynamic pre-aggs, tiny data, per-tenant/facility conditions). Cube.js wins if you later need ad-hoc analytical queries, dashboards, or data exploration.

Best strategy: Build the alert subscription engine as a custom service (BullMQ + Redis). Keep Cube.js for complex analytical queries that need a semantic layer. They can coexist — same MySQL sources, different purposes.
